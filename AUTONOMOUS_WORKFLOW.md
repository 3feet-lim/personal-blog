# Autonomous Workflow

## 목적

이 문서는 로컬 개발 환경에서 에이전트가 사람 개입 없이 연속적으로 작업할 때의 운영 원칙을 정의합니다.

대상 범위:

- 로컬 코드 구현
- 로컬 테스트와 검증
- 문서 갱신
- 기존 스펙 범위 안의 후속 작업

대상 제외:

- 운영 배포
- 외부 결제/과금 발생 작업
- 사용자 승인 없이 수행하면 위험한 파괴적 작업
- 시크릿이 없으면 진행할 수 없는 외부 연동 마무리

## 기본 원칙

- 중간 확인 없이 가능한 한 끝까지 진행합니다.
- 구현 중 발견한 작은 결정사항은 합리적인 기본값으로 스스로 확정합니다.
- 작업은 분석에서 멈추지 않고 구현, 검증, 수정, 문서화까지 이어서 수행합니다.
- 현재 저장소의 스펙과 이미 내려진 결정사항을 우선 기준으로 삼습니다.
- 기존 변경사항을 함부로 되돌리지 않습니다.
- 로컬 개발에서는 `수정 -> 빌드 -> 실행 -> 확인 -> 수정 반복`을 기본 루프로 삼습니다.

## 작업 시작 시 기본 해석

사용자가 기능 구현이나 개선을 요청하면 아래처럼 해석합니다.

- 별도 중단 지시가 없으면 다음 단계까지 연속 진행
- 사소한 애매함은 보수적인 가정으로 해소
- 같은 범위 안의 테스트 실패는 직접 수정 후 재검증
- 관련 문서는 필요 시 함께 갱신
- 빌드나 실행이 실패하면 원인을 보고 직접 수정한 뒤 다시 시도

## 계속 진행해도 되는 경우

아래 상황에서는 별도 확인 없이 계속 진행합니다.

- 기존 스펙 안에서의 기능 추가 또는 보완
- 로컬 빌드, 실행, 테스트, 데이터 시드
- 작은 스키마 수정이나 코드 구조 정리
- 누락된 문서 추가
- 테스트 실패에 대한 후속 수정
- UI/UX 흐름의 경미한 개선
- 환경 변수 예시나 로컬 설정 문서 정리

## 멈춰야 하는 경우

아래 상황에서는 작업을 멈추고 사용자 확인이 필요합니다.

1. 데이터 유실 가능성이 있는 파괴적 작업
2. 외부 비밀값, 계정, 인증 정보가 반드시 필요한 작업
3. 운영 배포 또는 운영 리소스 변경
4. 결제/과금이 발생할 수 있는 외부 서비스 사용
5. 현재 스펙을 벗어나는 큰 제품 결정
6. 사용자 기존 변경과 직접 충돌하는 경우

## 로컬 작업 루프

로컬 개발에서 에이전트는 보통 아래 흐름을 반복합니다.

1. 현재 스펙과 코드 상태 확인
2. 구현 범위 확정
3. 코드 수정
4. 빌드 실행
5. 실행 또는 테스트
6. 실패 시 원인 확인
7. 코드 다시 수정
8. 빌드/실행/테스트 재시도
9. 동작이 안정되면 관련 문서 갱신
10. 남은 리스크와 미완료 항목 보고

핵심은 추상적인 단계명을 따로 두는 것이 아니라, 실제 로컬 작업 흐름을 끝까지 반복하는 것입니다.

- 빌드가 실패하면 수정 후 재빌드합니다.
- 실행이 실패하면 로그를 보고 수정 후 재실행합니다.
- 테스트가 실패하면 실패 원인을 고친 뒤 다시 실행합니다.
- 한 단계가 통과되면 다음 단계로 자동 진행합니다.

## 가정 처리 원칙

- 명시되지 않은 UI 세부사항은 기존 화면 스타일을 따릅니다.
- 권한 정책이 애매하면 더 보수적인 쪽으로 구현합니다.
- 저장소 구조가 애매하면 `frontend`, `backend`, 루트 런타임 분리를 유지합니다.
- 외부 연동이 미완성일 때는 로컬 대체 흐름이 있으면 먼저 유지합니다.

## 최종 보고 원칙

최종 보고에는 아래만 간결하게 포함합니다.

- 실제로 변경한 내용
- 검증한 내용
- 아직 못 한 일
- 다음으로 자연스럽게 이어질 작업

중간 단계마다 다음 진행 여부를 묻는 대신, 멈춤 조건에 해당하지 않으면 계속 진행합니다.
