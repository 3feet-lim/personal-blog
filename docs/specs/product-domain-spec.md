# 개인 블로그 제품 / 도메인 스펙

## 1. 범위

이 문서는 아래 두 가지 핵심 콘텐츠 영역을 가진 개인 블로그 플랫폼의 제품/도메인 스펙을 정의합니다.

- 공개 기술 블로그
- 인증 및 인가된 사용자만 접근할 수 있는 비공개 가족 앨범

시스템 구성은 다음을 전제로 합니다.

- 프런트엔드: Next.js
- 백엔드: Python 기반 API/서비스
- 로컬 환경: Docker Compose
- 운영 환경: Kubernetes 배포가 존재하지만, 인프라 배포 세부사항은 이 문서 범위에서 제외
- 데이터베이스:
  - 로컬 개발/테스트: SQLite
  - 운영: PostgreSQL
- 객체 저장소:
  - 로컬/기본값: MinIO
  - 향후 확장: Amazon S3 호환 저장소
- 인증:
  - 초기 제공자: Google SSO
  - 향후 확장: Naver 및 기타 OAuth 제공자

이 문서는 제품 구조, 도메인 경계, 접근 규칙, 확장 지점에 집중합니다. 최종 구현 세부사항을 강제하지는 않습니다.

## 2. 목표

### 핵심 목표

- 하나의 홈 화면에서 아래 두 영역을 명확하게 분리해 제공합니다.
  - Tech Blog
  - Family Album
- 기술 블로그 콘텐츠는 로그인 없이 공개 접근을 허용합니다.
- 가족 앨범은 인증과 인가를 모두 통과한 사용자만 접근할 수 있어야 합니다.
- 앨범 콘텐츠의 이미지/미디어 업로드는 추상화된 객체 저장소 계층을 통해 지원합니다.
- Google SSO로 시작하되, 인증 모델은 특정 제공자에 종속되지 않게 유지합니다.
- 로컬 개발 환경은 Docker Compose로 가볍고 재현 가능해야 합니다.
- 운영 환경에서는 PostgreSQL과 S3 호환 저장소를 도메인 재설계 없이 사용할 수 있어야 합니다.

### 부가 목표

- 공개 콘텐츠와 비공개 가족 콘텐츠를 명확히 분리합니다.
- 아래 확장 가능성을 열어둡니다.
  - 추가 OAuth 제공자
  - 추가 비공개 콘텐츠 도메인
  - MinIO에서 S3 호환 클라우드 저장소로의 전환

## 3. 비목표

- Kubernetes 매니페스트 설계 또는 운영 배포 토폴로지
- CI/CD 설계
- 최종 UI 디자인 시스템
- 고급 편집 워크플로우
- 댓글, 좋아요, 피드 같은 완전한 소셜 기능
- 세밀한 엔터프라이즈 IAM 또는 멀티테넌트 SaaS 시나리오

## 4. 사용자 유형

### 4.1 사이트 방문자

공개 익명 방문자는 다음을 할 수 있습니다.

- 랜딩 페이지 접근
- 기술 블로그 글 읽기
- 태그, 카테고리, 글 목록 같은 공개 블로그 메타데이터 조회

다음은 할 수 없습니다.

- 가족 앨범 페이지 및 자산 접근
- 콘텐츠 업로드
- 보호된 API 접근

### 4.2 사이트 소유자 / 관리자

주요 운영자는 다음을 할 수 있습니다.

- SSO로 로그인
- 기술 블로그 콘텐츠 생성, 수정, 발행, 보관, 삭제
- 가족 앨범 컬렉션 생성 및 미디어 업로드
- 비공개 가족 콘텐츠 접근 권한 관리
- 허용 사용자 또는 초대 관리

### 4.3 승인된 가족 구성원

가족 앨범 영역 접근 권한이 부여된 인증 사용자입니다.

가능한 작업:

- 지원되는 OAuth 제공자로 로그인
- 권한이 허용된 가족 앨범 페이지와 미디어 접근
- 허용된 앨범과 이미지 조회

기본적으로 불가능한 작업:

- 관리자 기능 접근
- 별도 권한이 부여되지 않은 상태에서의 수정 또는 업로드

## 5. 제품 구조 / 정보 구조

### 5.1 최상위 영역

- `/`
  - 랜딩 페이지
  - 다음 영역으로 이동하는 주요 내비게이션
    - Tech Blog
    - Family Album
- `/blog`
  - 기술 블로그 목록
- `/blog/[slug]`
  - 블로그 글 상세
- `/album`
  - 가족 앨범 랜딩 페이지
  - 인증과 인가 필요
- `/album/[albumSlug]`
  - 앨범 상세 페이지
  - 인증과 인가 필요
- `/login`
  - 인증 진입점
- `/auth/callback/[provider]`
  - OAuth 콜백 처리
- `/admin`
  - 소유자/관리자용 콘텐츠 관리 영역
- `/admin/blog`
  - 블로그 관리
- `/admin/albums`
  - 앨범 관리
- `/admin/users`
  - 사용자/접근 관리

### 5.2 내비게이션 모델

랜딩 페이지는 아래 두 경로를 명확히 구분해서 보여줘야 합니다.

- Tech Blog: 공개 영역
- Family Album: 비공개 보호 영역

이 분리는 로그인 전에 이해 가능해야 합니다. 사용자는 앨범 접근에 로그인과 권한이 필요하다는 점을 미리 알아야 합니다.

## 6. 접근 모델과 인증 경계

### 6.1 인증

인증은 OAuth/OIDC 계열 외부 인증 제공자를 기준으로 설계합니다.

초기 제공자:

- Google

향후 제공자:

- Naver
- 기타 OAuth/OIDC 호환 제공자

내부 인증 모델은 정규화된 식별 계약을 넘어서 제공자별 필드에 의존하면 안 됩니다.

### 6.2 인가

가족 앨범 접근에는 인증만으로 충분하지 않습니다.

인가 수준은 최소한 아래를 구분해야 합니다.

- 익명 사용자
- 인증된 사용자
- 승인된 가족 사용자
- 관리자/소유자

접근 규칙:

- 기술 블로그:
  - 읽기: 공개
  - 작성/관리: 관리자만
- 가족 앨범:
  - 읽기: 인증 + 명시적 권한 필요
  - 작성/관리: 기본적으로 관리자만
- 관리자 영역:
  - 관리자만

### 6.3 자산 접근 경계

보호된 앨범 자산은 공개 정적 파일처럼 취급하면 안 됩니다.

저장소/접근 모델은 아래를 지원해야 합니다.

- 가족 미디어용 비공개 객체 저장 경로
- 자산 접근 전 애플리케이션 수준의 인가 검사
- 필요 시 향후 서명 URL 또는 임시 접근 토큰 지원

공개 블로그 자산은 별도로 저장하고 선택적으로 공개 제공할 수 있습니다.

## 7. 도메인 모델 초안

### 7.1 핵심 엔터티

#### User

내부 사용자 레코드를 나타냅니다.

필드:

- `id`
- `email`
- `display_name`
- `status`
- `role`
- `created_at`
- `updated_at`

비고:

- 한 사용자는 하나 이상의 인증 식별자를 연결할 수 있습니다.
- 역할 정의는 최소하고 명시적이어야 합니다.

#### AuthIdentity

연결된 외부 로그인 식별자를 나타냅니다.

필드:

- `id`
- `user_id`
- `provider`
- `provider_user_id`
- `provider_email`
- `provider_metadata`
- `last_login_at`

제약:

- `(provider, provider_user_id)` 조합은 유일해야 합니다.

#### Role / AccessGrant

보호 영역에 대한 인가를 나타냅니다.

가능한 형태:

- `User`의 단순 역할 필드
- 도메인별 권한을 위한 별도 접근 부여 테이블

권장 초기 모델:

- `User.role`: `admin | member`
- 필요 시 이후 앨범/도메인별 권한을 위한 `FamilyAccessGrant`

#### BlogPost

필드:

- `id`
- `slug`
- `title`
- `summary`
- `content`
- `status` (`draft | published | archived`)
- `published_at`
- `author_id`
- `created_at`
- `updated_at`

향후 선택 필드:

- `tags`
- `seo_title`
- `seo_description`
- `cover_image_asset_id`

#### Album

필드:

- `id`
- `slug`
- `title`
- `description`
- `visibility` (초기에는 항상 보호/비공개)
- `created_by`
- `created_at`
- `updated_at`

#### AlbumItem

앨범 안의 개별 이미지/미디어 항목을 나타냅니다.

필드:

- `id`
- `album_id`
- `asset_id`
- `caption`
- `taken_at`
- `sort_order`
- `created_at`

#### Asset

도메인 사용처와 분리된 업로드 바이너리 콘텐츠를 나타냅니다.

필드:

- `id`
- `storage_provider`
- `bucket`
- `object_key`
- `mime_type`
- `size_bytes`
- `checksum`
- `width`
- `height`
- `visibility` (`public | private`)
- `created_at`

이 추상화는 MinIO와 S3의 상호 교체를 가능하게 합니다.

### 7.2 관계 요약

- `User` 1:N `AuthIdentity`
- `User` 1:N `BlogPost`
- `Album` 1:N `AlbumItem`
- `AlbumItem` N:1 `Asset`
- `Asset`는 이후 `BlogPost` 또는 다른 도메인에서도 재사용할 수 있습니다.

## 8. 저장 전략

### 8.1 관계형 데이터

로컬 개발/테스트:

- SQLite

운영:

- PostgreSQL

설계 제약:

- 핵심 도메인 로직에서 운영 전용 PostgreSQL 기능에 의존하지 않되, 꼭 필요하면 명확히 분리합니다.
- 가능한 한 이식 가능한 스키마와 쿼리 패턴을 사용합니다.
- 마이그레이션 도구는 개발 라이프사이클에서 SQLite와 PostgreSQL 모두를 대상으로 할 수 있어야 합니다.

### 8.2 객체 저장소

초기 기본값:

- MinIO

향후 호환 대상:

- S3 호환 저장소 추상화

저장 카테고리는 논리적으로 분리해야 합니다.

- 공개 블로그 자산
- 비공개 가족 앨범 자산

권장 객체 키 접두사:

- `blog/public/...`
- `album/private/...`

도메인 로직에 MinIO 전용 의미를 하드코딩하면 안 됩니다. 저장소 연산은 저장 인터페이스/어댑터 경계를 통해 흘러야 합니다.

## 9. 로컬과 운영 환경 결정

### 9.1 로컬 개발

서비스 오케스트레이션은 Docker Compose를 사용합니다.

- 프런트엔드 컨테이너
- 백엔드 컨테이너
- 호환성 테스트용 선택형 PostgreSQL
- 가벼운 로컬 개발/테스트용 SQLite
- MinIO 컨테이너

로컬 우선순위:

- 빠른 기동
- 낮은 운영 복잡도
- 재현 가능한 개발 환경
- 클라우드 의존성 없이 인증/저장소 연동 테스트 가능

### 9.2 운영 전제

배포 세부사항은 범위 밖이지만, 도메인 전제는 다음과 같습니다.

- 주 관계형 데이터베이스는 PostgreSQL
- S3 호환 객체 저장소 사용 가능
- 애플리케이션 구성요소는 별도 컨테이너로 분리
- 인증 제공자 자격 증명은 외부에서 관리

## 10. 확장 지점

### 10.1 인증 제공자 확장성

인증 서브시스템은 아래를 포함하는 정규화된 제공자 계약을 정의해야 합니다.

- provider name
- external subject/user id
- email
- display name
- profile metadata

제공자별 매핑은 도메인 로직이 아니라 어댑터/연동 계층에 위치해야 합니다.

### 10.2 저장소 제공자 확장성

저장소 접근은 아래를 지원하는 추상화 뒤에 있어야 합니다.

- 업로드
- 삭제
- 메타데이터 조회
- 서명/비공개 접근 생성
- 필요한 경우 공개 URL 생성

이 구조를 통해 다음이 가능합니다.

- 지금은 MinIO
- 나중에는 S3
- 이후 다른 호환 제공자

### 10.3 콘텐츠 도메인 확장성

애플리케이션은 블로그 도메인과 앨범 도메인을 충분히 분리해, 향후 아래 같은 영역을 추가할 수 있어야 합니다.

- 메모
- 여행 기록
- 가족 공지
- 비공개 문서

## 12. API 초안

이 목록은 개념적 API 표면이며, 최종 구현 계약은 아닙니다.

### 12.1 인증

- `GET /auth/providers`
- `GET /auth/login/:provider`
- `GET /auth/callback/:provider`
- `POST /auth/logout`
- `GET /me`

### 12.2 블로그

- `GET /api/blog/posts`
- `GET /api/blog/posts/:slug`
- `POST /api/admin/blog/posts`
- `PATCH /api/admin/blog/posts/:id`
- `DELETE /api/admin/blog/posts/:id`

### 12.3 앨범

- `GET /api/albums`
- `GET /api/albums/:slug`
- `POST /api/admin/albums`
- `PATCH /api/admin/albums/:id`
- `POST /api/admin/albums/:id/items`
- `DELETE /api/admin/albums/:id/items/:itemId`

### 12.4 자산

- `POST /api/admin/assets/upload`
- `GET /api/assets/:id`
- `GET /api/assets/:id/access-url`

비고:

- 공개 블로그 자산은 직접 공개 URL로 해석될 수 있습니다.
- 비공개 앨범 자산은 URL 발급이나 스트리밍 전에 접근 검사를 거쳐야 합니다.

### 12.5 사용자/관리자

- `GET /api/admin/users`
- `PATCH /api/admin/users/:id`
- `POST /api/admin/access-grants`
- `DELETE /api/admin/access-grants/:id`

## 13. 예외 상황과 위험

- 사용자가 Google 인증에는 성공했지만 가족 콘텐츠 권한이 없는 경우
  - 기대 결과: 로그인은 성공, 앨범 접근은 거부
- 한 사람이 나중에 두 번째 제공자로 다시 로그인하는 경우
  - 식별자 연결 정책을 정의해야 함
- 제공자 이메일이 바뀌거나 제공되지 않는 경우
  - 내부 사용자 매핑은 변경 가능한 이메일 하나에만 의존하면 안 됨
- 비공개 이미지가 공개 객체 URL로 잘못 노출되는 경우
  - 공개/비공개 자산 경계가 명확해야 함
- SQLite와 PostgreSQL의 동작 차이
  - 스키마와 쿼리 설계는 차이를 최소화해야 함
- 큰 이미지 업로드
  - 제한, 검증, 저장 실패 처리 필요
- 앨범은 삭제되었지만 자산이 남는 경우
  - 자산 생명주기와 정리 정책을 정의해야 함
- 보호된 앨범 라우트나 미디어 URL에 대한 직접 접근 시도
  - 페이지와 자산 계층 모두에서 일관되게 거부되어야 함

## 14. 검증 계획

### 14.1 구현 전 검증

- 제품 구조와 접근 정책이 확정되어야 합니다.
- 공개 콘텐츠와 보호 콘텐츠의 경계가 명확해야 합니다.
- 인증 모델이 특정 제공자에 종속되지 않아야 합니다.

### 14.2 MVP 검증 기준

- 랜딩 페이지에서 블로그와 가족 앨범이 명확히 분리되어 보여야 합니다.
- 공개 블로그는 인증 없이 접근 가능해야 합니다.
- 가족 앨범은 승인된 사용자만 접근 가능해야 합니다.
- 관리자만 콘텐츠와 사용자를 관리할 수 있어야 합니다.
- 로컬 환경은 Docker Compose로 동작해야 합니다.
- 로컬 개발 경로는 SQLite에서 동작해야 합니다.
- 저장소 어댑터는 MinIO와 연동되어야 합니다.
- 저장소 계약은 MinIO 전용 동작을 가정하면 안 됩니다.
- 인증 계약은 도메인 재설계 없이 두 번째 제공자를 추가할 수 있어야 합니다.
- 보호 자산은 인가 없이 조회될 수 없어야 합니다.

### 14.3 회귀 중점 영역

고위험 영역:

- 인증 콜백과 세션 처리
- 역할/인가 검사
- 보호 자산 전달
- SQLite와 PostgreSQL 전제 사이의 데이터 이식성

## 15. 미해결 질문

- 가족 앨범 접근 대상은 정확히 누구인가?
  - 닫힌 허용 목록만 허용할 것인가?
  - 초대 기반 접근을 허용할 것인가?
- 사용자가 Google로 인증하면 자체 가입을 허용할 것인가, 아니면 모든 사용자가 사전 승인되어야 하는가?
- 가족 사용자는 조회 전용이어야 하는가, 아니면 일부 사용자에게 업로드/관리 권한을 줄 것인가?
- 블로그 글은 처음부터 임시 저장과 예약 발행을 지원해야 하는가?
- 블로그 글 작성 포맷은 Markdown으로 확정하는가?
- 비공개 앨범 이미지는 다운로드를 지원해야 하는가, 아니면 조회만 허용할 것인가?
- 초기 가족 앨범 범위에 동영상을 포함하는가, 아니면 이미지만 포함하는가?
- 앨범별 세부 권한이 필요한가, 아니면 공통 가족 권한 하나로 충분한가?
- 로컬 개발에서도 프런트엔드와 백엔드를 완전히 분리된 서비스로 배포해야 하는가?
- SQLite가 기본 경로이더라도, 로컬 Compose에 PostgreSQL을 호환성 테스트용으로 포함해야 하는가?
