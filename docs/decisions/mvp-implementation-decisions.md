# MVP 구현 결정사항

## 범위

이 문서는 개인 블로그 플랫폼의 MVP 구현을 시작하기 전에 필요한 제품/도메인 결정을 확정합니다.

참조 문서:

- `docs/specs/product-domain-spec.md`
- `tasks.md`

## 확정된 결정

### 1. 접근 정책

- 가족 앨범 접근은 허용 목록 기반으로 관리합니다.
- 인증만으로 가족 앨범 접근 권한이 생기지는 않습니다.
- MVP에서는 가족 영역 전체에 공통으로 적용되는 전역 권한을 사용합니다.
- 가족 사용자 기본 권한은 조회 전용입니다.
- 앨범 생성, 이미지 업로드, 콘텐츠 수정, 사용자 관리는 관리자만 수행할 수 있습니다.
- 인증은 되었지만 권한이 없는 사용자는 로그인 자체는 성공하되, 가족 라우트와 보호된 미디어에서는 접근 거부를 받아야 합니다.

### 2. 블로그 작성 포맷

- 블로그 글의 기본 작성 포맷은 Markdown입니다.
- Markdown 원문을 그대로 저장합니다.
- 렌더링 표현은 프런트엔드 책임입니다.
- 예약 발행, 협업 편집, 블록 에디터 같은 고급 작성 흐름은 MVP 범위에서 제외합니다.

### 3. 인증 정책

- 첫 번째 로그인 제공자는 Google입니다.
- 내부 사용자 식별 모델은 특정 제공자에 종속되지 않아야 합니다.
- 외부 인증 정보는 정규화된 내부 사용자 레코드로 매핑합니다.
- 사전 승인되지 않은 사용자의 자체 가입은 허용하지 않습니다.
- MVP에서는 내부 사용자 레코드가 이미 존재하는 상태를 사전 승인으로 간주합니다.
- Naver 및 추가 제공자는 확장 지점이며 MVP 필수 요구사항은 아닙니다.

### 4. 미디어 범위

- 가족 앨범은 이미지 중심으로 시작합니다.
- 동영상 업로드, 트랜스코딩, 재생은 MVP 범위에서 제외합니다.
- 지원 미디어 동작은 이미지 렌더링과 메타데이터 처리만 전제로 합니다.

### 5. 보호 자산 접근

- 가족 앨범의 보호 자산은 공개 정적 파일처럼 다루면 안 됩니다.
- 자산에 접근하려면 사용 가능한 URL이나 스트림을 발급하기 전에 애플리케이션 수준의 인가 검사를 거쳐야 합니다.
- 공개 블로그 자산과 보호된 가족 자산은 논리적으로 분리되어야 합니다.
- 로컬 객체 저장소는 MinIO를 사용하되, 애플리케이션은 이후 S3 호환 저장소로 교체 가능한 추상화 계층을 통해 접근해야 합니다.

## MVP 사용자 흐름

### 공개 사용자 흐름

1. 익명 사용자가 `/`를 엽니다.
2. 사용자가 공개 기술 블로그로 이동합니다.
3. 로그인 없이 발행된 Markdown 기반 글을 읽습니다.

### 제한된 사용자 흐름

1. 사용자가 Google로 로그인합니다.
2. 시스템이 외부 인증 정보를 내부 사용자 레코드로 매핑합니다.
3. 사용자가 사전 승인되어 있고 가족 권한이 있으면 가족 앨범을 탐색할 수 있습니다.
4. 사용자가 사전 승인되지 않았거나 가족 권한이 없으면 로그인은 유지되지만, 가족 페이지와 보호 자산에서는 접근 거부를 받습니다.

### 관리자 흐름

1. 관리자가 Google로 로그인합니다.
2. 관리자가 관리자 영역으로 진입합니다.
3. 관리자가 블로그 글, 앨범, 보호 자산, 허용 사용자 목록을 관리합니다.

## 구현을 위한 도메인 규칙

- `User`는 승인과 인가의 내부 기준 주체입니다.
- `AuthIdentity`는 제공자별 인증 연동 정보를 저장합니다.
- `BlogPost.content`는 Markdown 원문입니다.
- `Album`은 MVP에서 비공개 전용입니다.
- `Asset.visibility`는 공개 저장과 비공개 저장 동작을 구분해야 합니다.
- 객체 저장소가 나중에 서명 URL을 사용하더라도, 보호 자산 접근은 반드시 정책 검사를 거쳐야 합니다.

## 잔여 위험

- Google OAuth는 비밀값 없이도 골격 구현은 가능하지만, 전체 콜백 검증은 실제 환경 변수 설정이 필요합니다.
- SQLite와 PostgreSQL은 동작 차이가 있으므로 스키마와 쿼리 설계는 이식성을 유지해야 합니다.
- 앨범별 세부 권한은 과도한 설계를 피하기 위해 MVP 이후로 미룹니다.

## 즉시 구현 가이드

- 백엔드는 승인 여부와 가족 접근 권한을 분리해서 강제해야 합니다.
- 프런트엔드는 사용자가 가족 영역에 들어가기 전에 해당 영역이 보호 구역임을 명확히 보여줘야 합니다.
- 로컬 런타임은 SQLite 우선 기동을 기본으로 하고, 선택적으로 PostgreSQL 호환성 테스트를 지원해야 합니다.
- 저장소 연동은 MinIO 세부 구현을 어댑터 경계 뒤로 감춰야 합니다.
